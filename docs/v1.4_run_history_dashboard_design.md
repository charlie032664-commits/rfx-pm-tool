# v1.4 — Run History / Job Dashboard (Design)

Goal: a UI dashboard showing recent case runs at a glance, so PMs can see what
ran, its status, and whether outputs are ready — including stale jobs.

## Data sources (per case under runs/<case>/)

| File | Provides | Notes |
|------|----------|-------|
| `job_status.json` | current/last job: status, stage, provider, model, started/ended, pid, log_path | **primary** source; one object |
| `jobs.jsonl` | append-only history of job snapshots (start + terminal) | for a per-case timeline |
| `run_history.jsonl` | UI-path run records (start/terminal, steps, duration) | written by synchronous Streamlit runs only |

A run's outputs availability = which of `requirements.json`,
`requirements_clean.json`, `requirements_review.xlsx`, `compliance_matrix.xlsx`
exist in the run dir.

## CLI MVP (shipped in this branch)

`scripts/list_run_history.py` (read-only, stats only) already provides the data
layer:
```
python scripts/list_run_history.py --runs runs --limit 20      # table
python scripts/list_run_history.py --runs runs --json          # machine-readable
```
Columns: case_id, status, stage, provider/model, started, duration, outputs (N/4).
It scans `runs/**/job_status.json` (so sandbox runs appear too) and sorts newest
first. No LLM, no requirement text.

## UI panel design (v1.4)

- A "Run History" expander (or page) that calls `list_run_history.collect(RUNS_DIR)`
  and renders a `st.dataframe` with columns above, newest first.
- Row affordances (read-only first): show `log_path`, link to the case, and a
  **stale** badge when `status=running` but the pid is not alive (reuse
  `app._job_active` / `_pid_alive`).
- Optional filters: by case, by status. Default limit 20.
- No destructive actions in v1: surfacing only. (Cancel/clear already exist in the
  pipeline section.)

## Columns shown

case name · provider/model · status · stage · started/ended · duration · log path
· output availability (count present / 4).

## Risks / notes
- `run_history.jsonl` only covers UI-path runs; background-worker runs are in
  `job_status.json`/`jobs.jsonl`. The dashboard should prefer `job_status.json`
  (covers both) and treat `run_history.jsonl` as supplemental.
- Reading many files each render is cheap (small JSON) but could be cached with
  `st.cache_data(ttl=…)` if needed.
- Stale detection is display-only; never auto-delete.
