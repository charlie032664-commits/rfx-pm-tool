# v1.2 — Robust Background Pipeline Jobs (Design)

Status: design + initial implementation (feature/v1.2-pipeline-robustness)
Scope: make long pipeline runs survivable and observable. Initial implementation
is additive and does NOT refactor the existing CLI scripts.

## Current problem

The Full Pipeline is launched from `app.py` as a chain of subprocesses
(`extract_requirements_llm.py` → `run_case.py` → `postprocess_requirements.py`
→ `export_excel.py`), streamed line-by-line inside a single Streamlit script
run (`_stream` → `subprocess.Popen(... stdout=PIPE)`), guarded by a
`runs/<case>/.pipeline.lock` and logged to `runs/<case>/run_history.jsonl`.

Failure modes observed:
- **Streamlit rerun abandons the run.** Because the pipeline executes inside the
  Streamlit script run, any rerun (browser refresh, websocket reconnect, widget
  interaction) tears down the script thread. We have observed runs that stopped
  mid-Extract with the lock cleared but **no terminal `run_history` entry** and
  no final `requirements.json` — an ambiguous "did it finish?" state.
- **No single source of truth for "what is running now."** `.pipeline.lock` has
  liveness but no stage/provider/model; `run_history.jsonl` has history but is
  only written at start/end of a whole pipeline, not per stage.
- **CLI runs are invisible to the UI.** A pipeline run started from the command
  line writes nothing to `run_history.jsonl`.
- **Provider/model not recorded per run**, so a mixed-provider `--resume` can't
  be detected after the fact.

## Proposed background job status schema

A single JSON document per case: `runs/<case>/job_status.json` (current/last
job), plus an append-only `runs/<case>/jobs.jsonl` (history). Written by a small
shared helper `scripts/job_status.py` so both the UI and CLI can use it.

```json
{
  "job_id": "a1b2c3d4e5f6",
  "case_id": "2018_SilverPeak_EC-XS",
  "provider": "internal",
  "model": "qwen3-coder-next",
  "stage": "enrich",                 // extract | enrich | format | export | done
  "status": "running",               // queued | running | success | failed | cancelled
  "started_at": "2026-06-12T12:47:26",
  "ended_at": null,
  "pid": 13020,
  "log_path": "runs/<case>/_pipeline.log",
  "error": null,                     // short masked message on failure
  "updated_at": "2026-06-12T12:55:01"
}
```

Field notes:
- `job_id` — random hex (uuid4), stable for the whole pipeline run.
- `provider`/`model` — captured at start from `env_loader.describe_llm_config()`
  / `llm_client.get_model()` so a run's provenance is permanent.
- `stage` — advances as each step starts; lets the UI show "Enrich 3/4".
- `status` — terminal states are written even on failure/cancel so the
  "did it finish?" ambiguity disappears.
- `error` — short, secret-masked (never the API key).

## How the UI should start / monitor jobs

Start (future wiring; not in this initial commit):
1. UI acquires the existing `.pipeline.lock` (unchanged).
2. UI calls `job_status.start_job(run_dir, case_id, provider, model, log_path)`
   → writes `job_status.json` (status=running, stage=extract) and a `jobs.jsonl`
   line.
3. As each stage begins, `job_status.set_stage(run_dir, job_id, stage)`.
4. On completion/failure, `job_status.finish_job(run_dir, job_id, status, error)`.

Monitor:
- The UI reads `job_status.json` on each render (cheap) and shows
  stage/status/provider/model + elapsed. This is a *read* of persisted state, so
  it survives reruns: a refresh re-reads the file rather than losing the run.

## How to avoid Streamlit refresh interruption

Direction (incremental; only the persistence layer lands now):
- **Detach the worker from the script run.** Launch the pipeline as a detached
  child (e.g. `start`/`CREATE_NEW_PROCESS_GROUP`) that writes `job_status.json`
  + the log itself, instead of streaming inside the Streamlit run. Streamlit then
  only *polls* `job_status.json` — a rerun no longer kills the worker.
- Until detachment lands, `job_status.json` already removes the *ambiguity*: even
  if a streamed run is abandoned, the last persisted stage/status is visible, and
  a resume can be offered.
- `runOnSave` stays **false** so a code edit never auto-reruns a session.

## Validation plan

- Unit: `python scripts/job_status.py` self-test (create → set_stage → finish →
  read), in a temp dir, asserting schema + terminal states. No LLM, no network.
- `python -m py_compile app.py scripts/*.py`.
- Provider status: `python scripts/env_loader.py` (presence only).
- Smoke: `python scripts/llm_smoke_test.py` (tiny OK + synthetic JSON extraction).
- No full pipeline in this branch (especially not SilverPeak).

## Risks

- **Editing `app.py` while a run is active.** Mitigated by `runOnSave=false`
  (no auto-rerun) and additive, py-compiled edits.
- **Schema drift vs `run_history.jsonl`.** `job_status` complements (does not
  replace) run history; wiring is deferred so nothing breaks now.
- **Detachment portability (Windows).** Detached-process launch needs care with
  console handles / PID tracking; deferred to a dedicated change with its own
  validation.
- **Concurrency.** One job per case assumed (the existing lock enforces this);
  `job_status` does not add new locking.
