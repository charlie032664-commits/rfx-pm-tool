# v1.4 — Runtime Confirmation Gate (Design)

Goal: prevent accidental multi-hour internal runs while keeping small cases
frictionless. Builds on the v1.3 estimator (`scripts/runtime_estimator.py` +
the "⏱ Runtime estimate / large-run guard" expander in `app.py`).

## Current state (v1.3)

The expander has an **Estimate runtime** button that shows files / chunks /
runtime / risk and a warning for `warning`/`high`. It is **informational only** —
it does not gate the launch buttons.

## Proposed gate (by risk level)

`runtime_estimator.classify_risk(seconds)` → `ok` / `warning` / `high`
(thresholds: >1h warning, >4h strong warning).

| Risk | Behavior |
|------|----------|
| `ok` (<1h) | No gate. Buttons enabled as today. |
| `warning` (1–4h) | Show a yellow warning above the launch buttons; **buttons stay enabled** (no block). |
| `high` (>4h) | **Gate:** require an explicit confirmation before Full Pipeline / background-launch enable. |

### High-risk gate UX
- A checkbox: *"I understand this is estimated at ~Xh on `<provider>` and want to
  proceed."* — launch buttons disabled until checked.
- For the strongest case (internal **and** >4h), additionally require typing the
  case id (or the literal `RUN LARGE`) to enable — prevents a stray click.
- The estimate is computed on demand (reads files, no LLM); cache the result in
  `st.session_state` per case so the gate doesn't re-read files each render.

## Reuse, don't duplicate

`app.py` already has a **runtime-guard** section (Phase 4.6H) and a confirmation
checkbox `_confirm_full` driven by `req_json_exists` / `_est_chunks > 300`. The
v1.4 gate should **fold the estimator's risk into `_need_confirm`**:
```python
# pseudo
est = runtime_estimator.estimate_case(...)          # cached in session_state
_need_confirm = _need_confirm or est["risk"] in ("high",)
# warning risk -> show banner, do not force confirm
```
So one confirmation path, not a second competing gate.

## Constraints

- **Never block small/OK cases** — gate applies only at `high`.
- The gate is advisory + friction, not a hard prohibition; override is always
  possible via explicit confirmation (no silent block).
- Estimate is extract-only (enrich adds per-requirement) — label it as such so the
  number isn't mistaken for total wall-clock.

## Risk / decision
- Low-risk to implement (additive UI logic around existing buttons), but it
  touches the launch-button enable conditions — must be carefully tested so it
  never *disables* a small/OK case. Recommend implementing behind the existing
  `_confirm_full` plumbing with thorough manual UI testing. Document-first here;
  implement in a dedicated small PR.
