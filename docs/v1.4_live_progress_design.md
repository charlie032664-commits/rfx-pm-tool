# v1.4 — Live Pipeline Progress (Design)

Goal: show per-item progress (Enrich `item N/M`, Extract `chunk N/M`) for
**background worker** runs, matching what the synchronous UI already shows.

## Where progress is printed today

- `scripts/extract_requirements_llm.py:1501` → `print("[PROGRESS] <file> chunk i/total")`
- `scripts/run_case.py:488` → `print("[PROGRESS] enrich item i/total req_id=…")`

## Why the synchronous UI shows it live

`app.py` runs each step via `run_step_streaming` (`subprocess.Popen`,
`iter(proc.stdout.readline, "")`) with **`PYTHONUNBUFFERED=1`** injected into the
child env (app.py:333). It parses each `[PROGRESS]` line into widget state, so the
UI updates per item.

## Why the background worker log is only stage-level / buffered

`scripts/pipeline_worker.py:254` runs each stage with
`subprocess.run(cmd, stdout=logf, stderr=STDOUT, env=os.environ.copy())`:

1. **No `PYTHONUNBUFFERED`** in the child env → the child's stdout is
   **block-buffered** when redirected to a file, so `[PROGRESS]` lines flush in
   bursts (often only at stage end). `pipeline_worker.log` mtime appears "stuck"
   mid-stage even while work proceeds.
2. **`job_status.json` is updated only at stage boundaries** (`set_stage` at the
   start of each stage). There is no item-level field, so the persistent status
   panel can show `stage=enrich` but not `47/145`.

(Confirmed in this session: a healthy ~45-min SilverPeak enrich showed a static
log mtime while the process was clearly live — block buffering, not a stall.)

## What is needed for live item progress

### Option A — flush the child (low-risk, do first)
Add `PYTHONUNBUFFERED=1` to the worker's subprocess env:
```python
env = {**os.environ, "PYTHONUNBUFFERED": "1"}
subprocess.run(cmd, stdout=logf, stderr=subprocess.STDOUT, env=env, ...)
```
Effect: `[PROGRESS]` lines land in `pipeline_worker.log` live. Any reader/UI can
tail the log. ~1-line change, no refactor.

### Option B — surface item progress in job_status (the real fix)
Have the worker **stream** the child output instead of handing the fd to the file:
- `Popen(..., stdout=PIPE, text=True, bufsize=1, env={…PYTHONUNBUFFERED=1})`
- `for line in iter(proc.stdout.readline, ""):` → write line to `logf` AND, when a
  line matches `[PROGRESS] … (item|chunk) (\d+)/(\d+)`, update a new
  `stage_progress` field in `job_status.json` (e.g. `"47/145"`), throttled to ~1
  write/sec to avoid churn.
- Schema add (additive): `job_status.stage_progress: "47/145"` + optional
  `stage_progress_pct`. The UI panel renders it when present.

This reuses the exact regex `app.py` already uses; risk is contained to the
worker's stage loop.

## Safest implementation approach (recommended sequence)

1. **A now** — add `PYTHONUNBUFFERED=1` to the worker subprocess env. Trivial,
   makes the log live; no schema change. (Could even land as a v1.3.x fix.)
2. **B in v1.4** — worker tees child lines and writes `stage_progress` to
   `job_status.json`; UI shows "Enrich 47/145". Keep `subprocess.run` fallback if
   streaming setup fails.
3. Do **not** touch `run_case.py` / `extract_requirements_llm.py` printing — it is
   already correct; only the worker's capture path changes.

## Risks
- Throttle `job_status` writes (item cadence ~20s on internal, so cheap; but guard
  anyway).
- Streaming changes the worker's exit-code handling slightly — must still capture
  `proc.wait()` returncode and preserve the existing lock/finish semantics.
- Keep changes additive; the synchronous path is untouched.
