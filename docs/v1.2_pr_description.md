# v1.2: Robust background pipeline runner and LLM status tooling

Base: `master` (5702cbb) ← Head: `feature/v1.2-pipeline-robustness`

## Summary

Makes long pipeline runs survivable and observable. The Full Pipeline can now be
launched as a **detached background worker** that is not torn down by a Streamlit
rerun/refresh, with **persistent per-case job status**, **provider/model
visibility**, a **safe LLM smoke test**, and **shared lock protection** so
background and synchronous runs are mutually exclusive. All changes are additive —
the existing synchronous pipeline remains the default and is unchanged in
behavior.

## Why this change

Previously the pipeline ran *inside* the Streamlit script run, so any rerun
(refresh, websocket reconnect, widget interaction) could abandon a long run —
leaving the lock cleared but no terminal status and no final `requirements.json`
("did it finish?" ambiguity). There was also no record of which provider/model a
run used, and no quick way to confirm the LLM was reachable. v1.2 addresses all
of these without touching the core extraction logic.

## Major changes

- **Detached pipeline worker** (`scripts/pipeline_worker.py`): runs the 4 stages
  (extract → enrich → format → export) as a standalone process, invoking the
  existing CLI scripts as subprocesses (no refactor of them). Writes
  `runs/<case>/job_status.json` + `jobs.jsonl` + `pipeline_worker.log`; returns
  non-zero on failure; `--dry-run` walks the status flow with no LLM.
- **Persistent job status** (`scripts/job_status.py`): `start_job` / `set_stage`
  / `finish_job` / `read_job_status`; schema = job_id, case_id, provider, model,
  stage, status, started_at, ended_at, pid, log_path, error (secret-masked),
  updated_at. Wired into the synchronous Streamlit pipeline too.
- **Streamlit additions** (`app.py`, all additive):
  - sidebar **LLM provider** status (provider/model/per-var presence/ready — no
    secret values),
  - **"Run Full Pipeline in background (beta)"** button launching the worker
    detached (`DETACHED_PROCESS | CREATE_NEW_PROCESS_GROUP`),
  - **"Pipeline job status (persistent)"** readback (survives reruns),
  - **stale job recovery guidance** (status=running but dead pid → read-only
    warning; no auto-delete).
- **Shared pipeline lock**: the worker honors the same `runs/<case>/.pipeline.lock`
  protocol as the synchronous run (acquire-before-stages, refuse if active,
  replace only stale, release in `finally`) → background and sync runs are
  mutually exclusive.
- **Safe LLM smoke test** (`scripts/llm_smoke_test.py`): tiny "OK" generation +
  synthetic JSON extraction; PASS/FAIL + runtime; no customer data, no pipeline.
- **Design doc**: `docs/v1.2_pipeline_robustness_design.md`.

## Security / privacy notes

- No secrets are printed or committed. `job_status` masks secret-looking values
  out of any error message; the smoke test masks errors; the provider status
  shows presence only.
- `.env` is unchanged and remains gitignored (only `.env.example` is tracked).
- `runs/` and `inbound/` (RFQ business data) remain gitignored; no customer
  content is added to the repo.

## Known limitations

- The detached worker uses `job_status=running` + live pid as its concurrency
  guard in addition to the shared `.pipeline.lock`; one job per case is assumed.
- True extraction-cache reuse is NOT part of this PR (cache remains
  report-only).
- The background button has been CLI-validated but not yet click-tested in a
  live Streamlit session (see below).

## Manual test still required (before merge)

One interactive Streamlit check:
1. Click **"Run Full Pipeline in background (beta)"** on a small case.
2. **Refresh the page.**
3. Confirm the job **continues** and the "Pipeline job status (persistent)"
   panel advances stage/status to completion.

## Rollback notes

- Purely additive: reverting the merge (or the 10 commits) restores the prior
  synchronous-only behavior. No data migration; `job_status.json` / `jobs.jsonl`
  / `pipeline_worker.log` live under gitignored `runs/` and can be ignored or
  deleted. The synchronous path was never removed.

## Suggested reviewer checklist

- [ ] Worker invokes existing CLI scripts unchanged (no core extraction refactor).
- [ ] Lock acquire/refuse/replace-stale/release matches `app.py`'s protocol.
- [ ] `job_status` never logs secret values (error masking).
- [ ] Synchronous "Run Full Pipeline" path is unchanged.
- [ ] Background button disabled while a lock/active job exists.
- [ ] Manual Streamlit refresh-survival test performed.
