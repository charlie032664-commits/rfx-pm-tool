# v1.4 PM Workflow — Internal Release Note

**Audience:** PMs, sales engineers, and internal stakeholders
**Status:** Released to `master`
**Scope:** PM-facing workflow usability + reviewed-requirements wiring into the compliance matrix export

---

## Summary

v1.4 reshapes the RFx tool around the way a Product Manager actually works an
RFQ: a clean, numbered, top-to-bottom flow from "drop the files in" to
"download the customer-ready compliance matrix." The headline change is that the
PM's own review edits now flow all the way through to the exported matrix, and
the matrix can be regenerated on demand — with **no LLM call and no full
pipeline run** — so a PM can iterate on wording and responses in seconds.

## PM workflow: before vs after

**Before v1.4**

- Steps were labeled with internal/engineering names that didn't map to PM intent.
- Compliance responses were filled at an awkward point in the flow.
- The matrix was produced as a side effect of running the pipeline — there was
  no clear "make the deliverable now" action.
- **Crucially:** the PM's Step 4 review edits were *not* used by the export. The
  matrix was built from the machine-cleaned requirements only, so manual review
  work didn't reach the customer-facing file.

**After v1.4**

- Every step is named for what the PM is trying to do.
- Filling compliance responses is **Step 5**, right before generating the matrix.
- **Step 6** is a single, explicit button that (re)builds the deliverable.
- The export now prefers the PM's **reviewed** requirements, so review edits are
  honored. Regenerating is instant — no LLM, no re-extraction.

## Final Step 1 → Step 7 flow

1. **Step 1 — Case & Source Files:** select/create a case and add the RFQ files.
2. **Step 2 — Pre-check:** validate inputs before spending any analysis effort.
3. **Step 3 — Analyze RFQ:** extract and clean requirements (the AI engine step).
4. **Step 4 — Review Requirements:** PM reviews, edits, recategorizes, and can
   delete rows. Saved as `requirements_reviewed.json`.
5. **Step 5 — Fill Compliance Responses:** PM (or owners) fill status, vendor
   comment, gap, etc. Saved per case in `responses/<case>/responses.json`.
6. **Step 6 — Generate Compliance Matrix:** one click builds
   `compliance_matrix.xlsx` from the latest requirement source + saved responses.
7. **Step 7 — Download Results:** download the finished `compliance_matrix.xlsx`.

---

## Reviewed requirements integration

Step 6 selects its requirement source by **priority — newest/most-reviewed first**:

1. `requirements_reviewed.json`  ← PM's Step 4 reviewed edits
2. `requirements_clean.json`     ← machine-cleaned output
3. `requirements_mock.json`      ← mock/sample fallback
4. `requirements.json`           ← raw extracted fallback

The UI tells the PM exactly which source is in use:

- *"Using reviewed requirements."*
- *"Using cleaned requirements."*
- *"Using fallback requirement source: `<filename>`."*

If no requirement source exists yet, the button is disabled and the UI shows:
*"Please analyze the RFQ first before generating the matrix."*

**Reviewed-schema handling (in `export_excel.py`):**

- Detected by `meta.schema_version == "v1.4_review_mvp"`.
- `id` maps to `req_id` (and `orig_req_id`).
- `requirement`, `category`, `must_level`, `owner` carry through.
- Rows flagged `deleted: true` are **skipped** (they never reach the matrix).
- `pm_comment` is an internal note and is **omitted** from the customer-facing matrix.
- Existing clean/enriched export paths are untouched.

---

## Compliance response integration

- Step 5 responses are saved to `responses/<case>/responses.json`, keyed by `req_id`.
- During export, responses are merged onto each requirement **by `req_id`** —
  the same key whether the source is reviewed or cleaned, because `id → req_id`.
- If a requirement has no saved response, its response fields stay blank.
- The response-merge logic (`_merge_responses`) is unchanged from prior versions.

---

## Generate Matrix button behavior

- **Action:** runs `export_excel.py` only — assembles the Excel from JSON.
- **Does NOT:** call an LLM, run extraction, or run the full pipeline.
- **Inputs:** the priority-selected requirement source + saved responses.
- **Output:** `runs/<case>/compliance_matrix.xlsx` (12 sheets: Compliance Matrix,
  By_Category_Summary, six big-category sheets, Glossary, Notes, Skipped, Excluded).
- **Safe to re-run:** regenerate as often as you like after editing reviews or
  responses; each run is fast and deterministic.

## Known limitations

- **`pm_comment` is internal-only** in v1.4 — it is intentionally not surfaced in
  the customer-facing matrix. A separate internal/PM view is future work.
- **Reviewed schema is MVP (`v1.4_review_mvp`)** — it carries the core fields
  (`id`, `requirement`, `category`, `must_level`, `owner`, `deleted`). Richer
  fields (risk tags, normalization metadata) are not yet part of the reviewed
  round-trip and fall back to defaults when exporting from a reviewed source.
- **Source selection is by fixed priority**, not by timestamp — if both reviewed
  and clean exist, reviewed always wins (which is the intended behavior).
- **Step 6 is manual** — there is no auto-regenerate on save; the PM clicks the
  button to refresh.
