# v1.4 Backlog (prioritized)

Candidate work after v1.2/v1.3. Prioritized; each item should ship as its own
small PR with validation.

## P1 — high value, well-scoped

1. **UI live progress for Enrich (per item).**
   Surface `[PROGRESS] enrich item N/M` live in the UI for both sync and
   background runs. For background runs, have the worker write a lightweight
   progress field into `job_status.json` (e.g. `stage_progress: "47/145"`) so the
   panel can show it without tailing the log. Fixes the current "log doesn't
   update live during enrich" visibility gap (child stdout is block-buffered).

2. **Runtime estimator confirmation gate.**
   When the estimate is HIGH (>4h), require an explicit confirmation checkbox
   before the (background or sync) launch button enables — reuse
   `runtime_estimator.classify_risk`. Prevents accidental multi-hour internal runs.

## P2 — useful, moderate effort

3. **Comparison report UI panel.**
   A Streamlit panel wrapping `compare_model_outputs.py`: pick two run dirs,
   render the markdown report inline (stats only). MVP already exists as CLI.

4. **Run history dashboard.**
   Aggregate `job_status.json` / `jobs.jsonl` / `run_history.jsonl` across cases
   into a sortable table (case, provider/model, stage, status, duration). Helps
   PMs see what ran and what's stale.

## P3 — larger / needs design gates

5. **Guarded true cache reuse.**
   Per `docs/cache_reuse_design.md`: add `doc_schema_hash` to the signature,
   build a reuse store, gate behind `--use-extraction-cache` (default OFF), and
   require a clean-vs-reuse `compare_model_outputs.py` zero-drift check (baseline:
   `docs/v1.3_atlasrfq_comparison.md`) before enabling. Correctness-sensitive.

6. **Export QA checklist.**
   Automated pre-delivery checks on `compliance_matrix.xlsx` (no empty required
   cells, no duplicate ids, expected sheet set, row count vs requirements) with a
   pass/fail summary in the UI before download.

## Notes
- Keep the synchronous path as a fallback until the background worker has had
  broader real-world use.
- Internal-provider performance (~20s/call) remains the main runtime constraint;
  consider concurrency/batching only with careful rate-limit handling.
