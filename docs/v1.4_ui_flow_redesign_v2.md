# v1.4 — PM-Oriented UI Flow Redesign (v2)

Design only. Defines the target PM workflow **before** any `app.py` change.
No code is modified by this document.

## 1. Current UI flow (as built)

Sidebar: case mode (Select/Create) + case picker + **LLM provider** status.
Main page (top→bottom):

- **Step 1 — Upload RFQ Files**
- **Step 1.5 — Review & Select RFQ Files** (include/exclude per file)
- **Step 2 — Run Pipeline**: Run Full Pipeline · Enrich+Format+Export · Reset
  Extract · Run in background (beta) · "⏱ Runtime estimate" expander · job-status
  panel · stale-job guidance
- **Step 3 — Download Results**: `compliance_matrix.xlsx`, `requirements_review.xlsx`,
  "Advanced Outputs", "Incremental Cache Summary"
- **Step 3.5 — Normalize Requirements** (optional)
- **Step 4 — Review & Fill** (responses manager)

The structure is **pipeline-stage oriented** (Extract/Enrich/Format/Export),
which mirrors the engine, not the PM's mental model.

## 2. Current pain points

- **Runtime estimate feels like an add-on**, not a required pre-check before a
  potentially multi-hour run.
- **Background run vs Full Pipeline relationship is unclear** — two buttons, no
  guidance on which to use when.
- **Extract / Enrich / Format / Export are too technical** for PM users.
- **Job status and run actions are not grouped** as one coherent flow.
- **Compare tools are advanced QA**, surfaced near the main flow.
- **`requirements_review.xlsx` is presented like a deliverable**, competing with
  the real deliverable (`compliance_matrix.xlsx`).
- **No Review/Edit/Delete/Add** of requirements before matrix generation — the PM
  can't correct the AI output before producing the customer-facing matrix.

## 3. Proposed PM main workflow

A 6-step, outcome-oriented flow (engine stages hidden behind these):

1. **Case & Files** — pick/create case, upload, include/exclude files. (= current
   Step 1 + 1.5)
2. **Pre-check** — runtime estimate + provider/model readiness + large-run guard,
   shown **before** analysis as a required-to-see step (not an expander add-on).
3. **Analyze RFQ** — run extraction+enrichment (the engine's Extract→Enrich), with
   a clear **AI** vs **Fast UI Test / No-AI** choice; progress + job status shown
   here as one block.
4. **Review / Edit Requirements** — an editable table; PM corrects before the
   matrix is built (see §7).
5. **Generate Compliance Matrix** — build `compliance_matrix.xlsx` **from the
   reviewed requirements** (the engine's Format→Export).
6. **Download Results** — the matrix as the primary deliverable; everything else
   demoted to Advanced.

## 4. Advanced Tools section (collapsed, off the main path)

- **Continue Previous Run** (resume / Enrich+Format+Export only)
- **Rebuild Matrix** (regenerate from existing reviewed requirements)
- **Compare AI Results** (`compare_model_outputs.py` UI)
- **Run History** (`list_run_history.py` dashboard)
- **Cache / Diagnostics** (Incremental Cache Summary, raw JSON, logs)
- **Model Settings** (provider/model presence — read-only)

## 5. AI vs Non-AI mapping

| Step | AI? |
|------|-----|
| 1. Case & Files | No AI |
| 2. Pre-check | Mostly No AI (estimator reads files; a smoke test may call LLM only if the PM clicks it) |
| 3. Analyze RFQ | **AI**, or **No-AI Flow Test** (explicit choice) |
| 4. Review / Edit Requirements | No AI |
| 5. Generate Matrix | No AI |
| 6. Download Results | No AI |
| Advanced (Compare / Run History / Cache) | No AI unless explicitly requested |

## 6. No-AI Flow Test placement

- Lives **under Analyze RFQ** as **"Fast UI Test / No-AI"** (wraps
  `scripts/mock_extract_requirements.py`).
- For **workflow testing only** — lets a PM click through the whole flow without
  spending LLM time/budget.
- Must be clearly labeled **not final RFQ extraction quality**; its output should
  be visibly marked `analysis_mode = no_ai_mock` and never presented as a real
  analysis.

## 7. Requirement review principle

Before final matrix generation, the PM must be able to:

- edit requirement text,
- delete an incorrect item,
- add a missing item,
- change category,
- change must / optional / info (must-level),
- add a comment,
- **save** the reviewed requirements (a stable, PM-owned artifact).

Suggested store: `runs/<case>/requirements_reviewed.json` (PM-edited, separate
from the AI's `requirements_clean.json`), so re-analysis never silently
overwrites PM edits. The matrix is generated from this when present.

## 8. Output principle

- **`compliance_matrix.xlsx` is the deliverable** and is generated from the
  **PM-reviewed** requirements.
- **`requirements_review.xlsx` is advanced/debug only** — moved under Advanced.
- Raw JSON / logs / cache details stay under **Advanced Tools**, not the main flow.

## 9. Recommended implementation phases

- **Phase A — Reorder UI sections only** (no behavior change): regroup existing
  widgets into the 6-step flow + Advanced Tools. Lowest risk.
- **Phase B — Add No-AI Flow Test entry** under Analyze RFQ (wrap the existing
  mock CLI).
- **Phase C — Add Review/Edit requirements table** (`st.data_editor`) + save to
  `requirements_reviewed.json`.
- **Phase D — Generate matrix from reviewed requirements** (prefer
  `requirements_reviewed.json` when present; fall back to `requirements_clean.json`).
- **Phase E — Download results cleanup** (matrix primary; review.xlsx demoted).
- **Phase F — Advanced tools cleanup** (Compare / Run History / Cache / Model
  settings consolidated).

Each phase ships as its own small PR with manual UI testing; the synchronous and
background pipeline behavior underneath stays unchanged.

## 10. Risk assessment

| Change | Risk |
|--------|------|
| This doc (docs only) | **Low** |
| UI reorder (Phase A) | Medium-low (layout only; test no widget keys collide) |
| Review/Edit data model (Phase C) | Medium (new `requirements_reviewed.json` schema + editor state) |
| Matrix from reviewed requirements (Phase D) | Medium (export must read the reviewed source correctly; keep fallback) |
| Cache reuse | **High — should wait** (see `docs/v1.4_cache_reuse_roadmap.md`; gated by zero-drift) |
