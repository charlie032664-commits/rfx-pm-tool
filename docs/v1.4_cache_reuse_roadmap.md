# v1.4 — Guarded Extraction-Cache Reuse (Implementation Roadmap)

Builds on `docs/cache_reuse_design.md`. This is a **roadmap only** — do not
implement true reuse until the validation gate (below) passes. Default OFF.

## Today (recap)
- `scripts/extraction_cache.py` = metadata + `should_skip` + `CacheSummary`.
- `scripts/cache_metadata.py` = report-only; writes `runs/<case>/extraction_cache.json`.
- `scripts/extract_requirements_llm.py` does the real extraction; cache is NOT
  consulted during extraction. No reuse happens.

- **A** must run *after* the `--resume` skip and *before* `call_llm_json_with_retry`.
- **B** must run *after* the req_id/source fix-ups so the cached objects are the
  final shape.
- The reuse store is a new sidecar, e.g. `runs/<case>/extraction_reuse.json`
  (signature → requirement objects). It is NOT partial.jsonl (different keying).

## Required cache signature (sha256 over)

| Component | Source | Why |
|-----------|--------|-----|
| file hash | manifest sha256 of the source file | source bytes changed → miss |
| chunk text hash | sha256 of the chunk string | the actual unit changed → miss |
| prompt version | `extraction_cache.PROMPT_VERSION` | prompt change → miss |
| model | resolved `get_model()` | model swap → miss |
| **schema version / doc_schema hash** | sha256 of `inbound/<case>/meta/doc_schema.json` | **NEW — req_id rules / routing change → miss** |
| parser version | `extraction_cache.PARSER_VERSION` | block/chunk logic change → miss |
| extraction_config | canonical `{max_chars, group_size, …}` | chunk boundaries change → miss |

`extraction_cache.cache_signature` already covers all but the **doc_schema hash**
— that must be added.

## Invalidation rules
- Any signature component differs → cache miss → re-extract (safe default).
- Missing/unreadable reuse store → treat as empty (all miss).
- `--reset-partial` or a new `--reset-cache` clears the reuse store.

## Correctness risks (must all be mitigated by the signature)
- Changed requirement text / source bytes → file+chunk hash.
- Changed prompt → prompt version.
- Changed model → model in signature.
- Changed chunking → extraction_config + parser version.
- Changed extraction schema / req_id rules → **doc_schema hash (new)**.
- Stale ids → cached objects keep ids verbatim; the existing `dedup_requirements`
  must still run after reuse so reused + fresh ids don't collide.

## Fallback behavior
- Reuse is best-effort: any error in lookup/store → fall through to a normal LLM
  call (never fail the run because of the cache).
- With the flag OFF (default), behavior is byte-identical to today.

## Validation strategy (gate before enabling)
1. Run AtlasRFQ clean (no cache) → baseline (already have
   `docs/v1.3_atlasrfq_comparison.md`).
2. Run AtlasRFQ with `--use-extraction-cache` against a prior identical run.
3. `compare_model_outputs.py --left <clean> --right <cached>` must show **zero
   drift** (counts, ids, distributions identical).
4. Only after AtlasRFQ passes: repeat on Nokia, then SilverPeak (large) — each
   must show zero drift before the flag is considered safe.

## CLI / UI design
- CLI: `--use-extraction-cache` (default **off**) on `extract_requirements_llm.py`;
  optional `--reset-cache`.
- UI: an unchecked-by-default "Reuse unchanged extractions (experimental)" toggle;
  caption must read "Experimental — reuses prior LLM extractions for unchanged
  chunks" and the cache summary must switch from the current
  "Report-only. No LLM work is skipped yet." to "Reuse ON — N chunks reused" only
  when the flag is active.

## Decision
Document-only for now. Implement in stages behind the flag, each gated by a
zero-drift comparison, starting with AtlasRFQ. Do not enable by default until
SilverPeak-scale zero-drift is demonstrated.
